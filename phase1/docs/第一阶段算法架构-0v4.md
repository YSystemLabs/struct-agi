# 第一阶段算法架构

> 版本：v0.4
>
>状态：使用中（Step 2 已验收冻结，Step 3 准备阶段）

## 1. 文档定位

本文档是第一阶段研究计划（v0.1）的算法实施方案。研究计划定义了"做什么"和"为什么做"，本文档定义"怎么做"。

核心设计原则有两条：

> **原则一**：第一阶段不做大一统模型，只做**最小可证伪的结构学习闭环**。

闭环的含义是：网格 → 对象/关系/变换/约束 → 中间结构表示 → 答案 → 归因。每一步的输入输出都必须可审计。

> **原则二**：系统不是"每个任务从零开始"的无记忆流水线，而是维护一个**可增长、可重写的知识状态**，在任务序列中渐进积累结构知识。

这一原则来自一个基本洞察：开放扩展若只有"加法"而无"重写"，必然退化为规则堆积；扩展后触发保行为的重编译，才能将积累转化为系统级结构升级。

## 2. 总体架构

系统分为五层，各层职责、输入、输出和实现策略如下。

```text
┌─────────────────────────────────────────────────────────────┐
│  Layer 5 · 验证与归因层                                      │
│  输入：候选解 + 中间结构    输出：通过/失败 + 失败归因标签     │
├─────────────────────────────────────────────────────────────┤
│  Layer 4 · 求解层                                            │
│  输入：筛选后结构表示       输出：test output 网格            │
├─────────────────────────────────────────────────────────────┤
│  Layer 3 · 任务相关结构筛选层                                │
│  输入：候选结构集合          输出：最小充分结构子集            │
├─────────────────────────────────────────────────────────────┤
│  Layer 2 · 候选生成层                                        │
│  输入：对象图 × 示例对      输出：候选变换 + 候选约束          │
├─────────────────────────────────────────────────────────────┤
│  Layer 1 · 结构感知层                                        │
│  输入：原始网格              输出：多方案候选对象图            │
└─────────────────────────────────────────────────────────────┘
```

五层与研究计划中四模块的对应关系：

| 研究计划模块 | 架构层 |
| --- | --- |
| 模块一：结构候选生成 | Layer 1 + Layer 2 |
| 模块二：结构筛选 | Layer 3 |
| 模块三：保结构表示 | Layer 3 输出 + Layer 4 输入的接口格式 |
| 模块四：求解器 | Layer 4 |
| 失败归因框架（第10节） | Layer 5 |

## 3. Layer 1：结构感知层

### 3.1 职责

从原始input网格中提取候选对象和对象间关系，输出**多个候选分割方案**而非唯一分割。

### 3.2 为什么输出多方案

同一网格可以有多种合理分割——按单色连通块、按矩形区域、按重复图样、按背景/前景分离。哪种分割对当前任务最有用，感知层无法独立判断，必须让 Layer 3 后验决定。因此 Layer 1 的输出是一个候选分割方案的集合。

### 3.3 对象提取策略

采用强启发式打底，不引入学习。ARC 的离散网格世界里，以下提取近乎零成本且准确率极高：

| 策略 | method 标识 | 方法 | 提取目标 |
| --- | --- | --- | --- |
| 4-连通分量 | cc4 | 按颜色做 4-连通 BFS/DFS | 基础色块（严格邻接） |
| 8-连通分量 | cc8 | 按颜色做 8-连通 BFS/DFS | 基础色块（含对角邻接） |
| 矩形区域检测 | bbox | 扫描单色或多色填充的矩形边界 | 规则区域 |
| 背景/前景分离 | bg_fg | 频率最高的颜色作为候选背景色 | 信号/噪声区分 |
| 重复图样检测 | repeat | 滑窗 + 哈希匹配，检测平移周期性 | 重复单元 |
| 层级对象 | nested | 大区域内部递归提取子连通分量 | 嵌套结构 |
| 整体网格 | whole_grid | 把整个网格视为单一对象，不做分割 | 网格级变换 |

cc4 与 cc8 是**两个独立策略**，各自产出一个分割方案。原因：在 ARC 任务中，对角邻接的语义差异显著——4-连通倾向于将对角接触的色块分开（适合 Copy、MoveToBoundary 等需要精确定位的任务），8-连通倾向于将它们合并（适合 CompleteShape、FilledNotFilled 等需要完整形状的任务）。将两者作为同一策略的变体会丢失这层区分。

共 7 种策略，每种独立运行，各产出一个分割方案。所有方案打包输出。

**关于"整体网格"方案**：部分 ARC 任务的规则作用于整个网格而非对象集合（如"输出 = 输入旋转 90°"，或 `007bbfb7` 式的网格自乘积）。在这些任务上，强制进行对象分解反而会引入噪声。整体网格方案把网格本身作为唯一对象，Layer 2 的部分原语（rotate、flip、tile、crop）可直接作用于它。

### 3.4 关系提取

在每个分割方案内，对所有对象对提取以下关系：

**第一阶段必须支持的关系类型（初始集）：**

| 关系类型 | 说明 |
| --- | --- |
| 邻接 | 两对象是否共享边界像素 |
| 包含 | 一个对象的 bbox 是否完全落在另一个内部 |
| 对称 | 两对象是否关于某轴或某点对称 |
| 相对位置 | 上下左右、对角方位 |
| 颜色映射 | 两对象颜色是否存在一一映射或常量偏移 |
| 尺寸映射 | 两对象的行数/列数是否存在整数比或相等关系 |
| 对齐 | 两对象是否共享行/列/中心线 |

**预留扩展但第一阶段不强制实现的关系类型：**

| 关系类型 | 说明 |
| --- | --- |
| 条件依赖 | 对象属性决定其应接受的变换类型 |
| 计数/奇偶性 | 某类对象的数量 → 影响输出结构 |
| 层级归属 | 对象属于哪个父级结构 |

关系类型表设计为可扩展的注册表，新关系类型通过实现统一接口加入。每次有任务因关系类型不够而归因失败时，记录到扩展候选清单。

### 3.5 输出格式

```text
PerceptionOutput = {
  segmentation_plans: [SegmentationPlan],   -- 多个候选分割方案
}

SegmentationPlan = {
  plan_id: str,
  method: str,                              -- "cc4" | "cc8" | "bbox" | "bg_fg" | "repeat" | "nested" | "whole_grid"
  objects: [Object],
  relations: [(ObjectID, ObjectID, Relation)],
}

Object = {
  id: ObjectID,
  pixels: set[(row, col)],                  -- 像素坐标集合
  bbox: (min_row, min_col, max_row, max_col),
  color_hist: dict[int, int],               -- 颜色频率
  attrs: {                                  -- 可扩展属性字典
    dominant_color: int,
    is_rectangular: bool,
    area: int,
    ...
  },
}
```

## 4. Layer 2：候选生成层

### 4.1 职责

基于 Layer 1 输出的对象和关系，结合每个训练示例的 input-output 对，提出候选变换和候选约束。

### 4.2 变换原语库

采用**有限文法**策略：用少量原语的受控组合覆盖任务空间，而非开放式搜索。

#### 4.2.1 初始原语集

第一阶段从以下原语集启动：

| 类别 | 原语 | 签名（简写） |
| --- | --- | --- |
| 几何 | translate | Object × Direction × Distance → Object |
| 几何 | rotate | Object × Angle{90,180,270} → Object |
| 几何 | flip | Object × Axis{H,V} → Object |
| 几何 | scale | Object × Factor → Object |
| 拓扑 | copy | Object → Object × Object |
| 拓扑 | delete | Object → ∅ |
| 拓扑 | fill | Region × Color → Object |
| 拓扑 | extend_to_boundary | Object × Direction → Object |
| 映射 | recolor | Object × ColorMap → Object |
| 映射 | conditional_map | Object × Predicate × Transform → Object |
| 组合 | merge | [Object] → Object |
| 组合 | sort | [Object] × Key → [Object] |
| 组合 | group_by | [Object] × Attr → [[Object]] |
| 网格 | crop | Grid × Region → Grid |
| 网格 | tile | Grid × Factor → Grid |
| 网格 | construct_grid | [Object] × Layout → Grid |

初始 16 个原语。设计原则：宁少勿多，边跑任务边扩展。

**预留原语（Step 3+ 按需激活）：**

| 类别 | 原语 | 签名（简写） | 激活条件 |
| --- | --- | --- | --- |
| 组合 | partition | Object × Predicate → [Object] | §4.5 检测到 split_groups 且归因为 ABSTRACTION_FAIL |

`partition` 将一个对象按谓词拆分为多个子对象，是 `merge` 的逆操作。Step 1-2 不激活（仅支持 1:1 对象映射任务）；Step 3 起当 §4.5.2 的 `split_groups` 非空且归因指向"无法表达拆分"时，由 §11.3 的扩展准入机制将其注册到 $K_t$ 中。

**关于网格级原语**：约 30% 的 ARC 任务输入输出尺寸不同。`crop` 从网格中裁剪子区域作为输出（如 ExtractObjects 类任务，10×10 → 3×1）；`tile` 将网格按因子平铺（如 `007bbfb7`，3×3 → 9×9 的自乘积）；`construct_grid` 根据对象集合和布局规则构建一个新网格（如 Count 类任务将计数摘要输出为小网格）。

**关于 `copy` 的语义**：`copy` 定义为"保留原件 + 产生一个可被后续原语独立操作的副本"。`copy` 的输出类型是 `Object × Object`（原件, 副本），在 DSL 中通过以下机制解构：

- **焦点选择器**：`copy` 之后必须跟 `on_original:` 或 `on_copy:` 焦点选择器，指定后续变换作用于原件还是副本。
- **语法**："复制到某位置"写作 `copy ; on_copy: translate(dir, dist)`——先复制，再将焦点切换到副本并平移。原件保持原位。
- **完整执行语义**：`copy` 执行后，解释器将当前对象集合从 `{obj}` 更新为 `{obj_original, obj_copy}`（两者像素集合相同，ID 不同）。`on_original:` / `on_copy:` 选择其中一个作为后续变换的目标。若需对两者均做变换，写作 `copy ; on_copy: T1 ; on_original: T2`。
- **默认焦点**：若 `copy` 后不显式写焦点选择器，解释器报错（不允许歧义）。

这消除了"translate 作用于原件还是副本"的歧义，使 `copy` 在 DSL 中完全可执行。

#### 4.2.2 原语组合规则

- 组合深度限制：初始最大 3 层（即最多 3 个原语串联）。
- 组合方式：序列复合（A 然后 B）、条件分支（if P then A else B）、映射（对每个对象执行 A）。
- 类型检查：只有输出类型与下一步输入类型兼容的组合才合法。

#### 4.2.3 原语库扩展机制

当某个任务在 Layer 5 归因为"候选变换不足"时：

1. 人工分析该任务所需的变换；
2. 判断是否需要新原语，还是现有原语的新组合方式；
3. 若需新原语，实现并注册到原语表；
4. 记录扩展日志，追踪原语库的增长曲线。

### 4.3 输出尺寸推断

约 30% 的 ARC 任务输入输出网格尺寸不同。在进入候选变换搜索之前，先从 train pairs 归纳输出尺寸规律：

1. **等尺寸**：所有 train pair 的 input 和 output 尺寸相同 → 输出网格尺寸 = 输入网格尺寸。
2. **固定尺寸**：所有 train pair 的 output 尺寸相同（但与 input 不同）→ 输出网格为固定尺寸。
3. **比例关系**：output 尺寸是 input 尺寸的整数倍或整数分之一 → 推断比例因子。
4. **依赖对象属性**：output 尺寸与 input 中某类对象的数量、尺寸等属性相关 → 建立属性到尺寸的映射。

输出尺寸推断的结果会作为约束传递给 Layer 3 和 Layer 4，在求解时先确定目标网格的形状。

### 4.4 Diff 驱动的候选生成

Layer 2 的候选生成**不是盲枚举**，而是由 input-output 差异驱动。

对每个 train pair，在给定分割方案下：

1. **对象对齐**：将 input 图和 output 图的对象进行对齐（详见 4.5 节）。
2. **Diff 计算**：识别对齐后的差异类型，并将每类差异映射到候选原语族。

| 差异类型 | 判定信号 | 优先生成的候选原语 |
| --- | --- | --- |
| 新增对象 | output 中存在但 input 中无对应对象（`unmatched_output`） | copy / fill / construct_grid |
| 消失对象 | input 中存在但 output 中无对应对象（`unmatched_input`） | delete / crop |
| 对象合并 | 多个 input 对象对应一个 output 对象（`merge_groups`） | merge |
| 对象拆分 | 一个 input 对象对应多个 output 对象（`split_groups`） | partition（Step 3+ 激活） |
| 位置变化 | 对齐对象位置发生偏移 | translate（从偏移量反推参数） |
| 形状变化 | 对象像素集合改变但颜色不变 | rotate / flip / scale / extend_to_boundary |
| 颜色变化 | 对象形状不变但颜色改变 | recolor / conditional_map |
| 尺寸变化 | 网格整体大小发生变化 | crop / tile / construct_grid |

1. **反向推断**：对每种差异，反向推导能产生该差异的原语及其参数。仅对这些原语生成候选，而非枚举所有类型合法的程序。

这种 diff 驱动策略将搜索空间从"所有合法程序"缩小到"能解释观察到的差异的程序"，通常缩小若干个数量级。

### 4.5 Input-Output 对象对齐

对象对齐是 diff 驱动生成和约束提取的前提，也是整个系统中最困难的子问题之一。对齐必须处理的情况包括：对象数量不等（有新增/删除）、对象位置发生变化、多个对象合并为一个、一个对象拆分为多个。

采用**多策略候选 + 最优匹配**的方法：

#### 4.5.1 匹配策略

| 策略 | 方法 | 适用场景 |
| --- | --- | --- |
| 像素重合匹配 | 两对象像素集合的 IoU > 阈值 | 对象位置未变或微调 |
| 颜色+形状匹配 | 颜色直方图 + 形状描述符（面积、宽高比）距离 | 对象发生平移但保持形状 |
| 最优二部匹配 | 匈牙利算法，代价矩阵 = 多属性加权距离 | 一对一场景 |
| 合并检测 | 多个 input 对象的像素并集 ≈ 单个 output 对象 | many-to-one（对象合并） |
| 拆分检测 | 单个 input 对象的像素 ≈ 多个 output 对象的并集 | one-to-many（对象拆分） |

合并/拆分检测的具体方法：对 `unmatched_input` 中的对象子集，检查其像素并集与某个 `unmatched_output` 对象的 IoU；对称地检查拆分。枚举子集时限制最大组大小（初始 ≤ 4 个对象）以控制复杂度。

#### 4.5.2 对齐输出

```text
Alignment = {
  alignment_id: str,                        -- 稳定主键，被 CandidateTransform/CandidateConstraint 引用
  matched_pairs: [(InputObjectID, OutputObjectID, match_score)],
  merge_groups: [MergeGroup],               -- many-to-one：多个 input 对象合并为一个 output 对象
  split_groups: [SplitGroup],               -- one-to-many：一个 input 对象拆分为多个 output 对象
  unmatched_input: [InputObjectID],         -- input 中无对应的对象（被删除）
  unmatched_output: [OutputObjectID],       -- output 中无对应的对象（新增）
}

MergeGroup = {
  input_ids: [InputObjectID],               -- 参与合并的 input 对象（≥ 2）
  output_id: OutputObjectID,                -- 合并后的 output 对象
  merge_score: float,                       -- 像素并集的 IoU
}

SplitGroup = {
  input_id: InputObjectID,                  -- 被拆分的 input 对象
  output_ids: [OutputObjectID],             -- 拆分产生的 output 对象（≥ 2）
  split_score: float,                       -- 像素并集的 IoU
}
```

对每个分割方案，所有策略独立运行，输出多个候选对齐方案。Layer 3 在筛选时同时决定最优分割方案和最优对齐方案。

**Scope 说明**：Step 1-2 只支持一对一对象映射任务（`merge_groups` 和 `split_groups` 为空）。Step 3 起逐步支持 merge/split，优先处理 ConceptARC 中涉及合并/拆分的概念组。

### 4.6 约束提取

基于 4.5 节的对齐结果，对每个 train pair 检查以下候选约束：

| 约束类型 | 检查方法 |
| --- | --- |
| 计数守恒 | 某类对象在 input 和 output 中数量是否相等 |
| 颜色映射一致性 | 所有示例中同一颜色是否映射到同一目标颜色 |
| 相对位置不变 | 对象间的方位关系在变换前后是否保持 |
| 局部一致性 | 对象内部结构在变换后是否保持 |
| 输出格式约束 | 输出网格的尺寸、形状是否满足某种规律 |
| 全局单调性 | 某种属性（如面积、颜色数）是否在变换后单调变化 |

### 4.7 候选生成的输出

```text
CandidateSet = {
  plan_id: str,                             -- 来自哪个分割方案
  candidate_alignments: [Alignment],        -- 该分割方案下的候选对齐方案（来自 §4.5）
  candidate_transforms: [CandidateTransform],
  candidate_constraints: [CandidateConstraint],
}

CandidateTransform = {
  transform_id: str,
  alignment_id: str,                        -- 该变换来自哪个原始对齐方案
  alignment_family_id: str,                 -- 该变换绑定到哪个 family 级对齐键
  program: ProgramSketch,                   -- DSL 程序草图
  applicable_pairs: [int],                  -- 在哪些 train pair 上可执行
  match_score: float,                       -- 在可执行的 pair 上的匹配度
}

CandidateConstraint = {
  constraint_id: str,
  alignment_id: str,                        -- 该约束来自哪个原始对齐方案
  alignment_family_id: str,                 -- 该约束绑定到哪个 family 级对齐键
  predicate: ConstraintPredicate,
  holds_in: [int],                          -- 在哪些 train pair 上成立
}
```

`alignment_id` 使 Layer 3 在筛选时可以追溯变换和约束各自来源于哪个对齐方案，并确保同一假设内的程序和约束绑定到同一对齐（参见 §5.2 绑定规则）。Layer 5 归因时也可以将失败定位到具体对齐错误（例如"该对齐将两个不相关的对象配对"），而非笼统地归为 SELECTION_FAIL。

Step 1 实现注记：为跨多个 train pair 聚合同一类对齐证据，当前 Step 1 代码显式使用 `alignment_family_id = "{plan_id}:{method}"`。其中，原始 `Alignment.alignment_id` 仍是 pair 级 ID（`"{plan_id}:{method}:{pair_index}"`）；一旦进入跨 pair 聚合后的候选与假设装配阶段，raw provenance 继续保留在 `alignment_id`，family 级绑定键单独保留在 `alignment_family_id`。

### 4.8 ProgramSketch 的定义

`CandidateTransform.program` 的类型为 `ProgramSketch`，它是 §6.5 完整 `Program` 语法的一个**受限子集**，由 Layer 2 的 diff 驱动反向推断自动生成。

```text
ProgramSketch ::= SketchStep | ProgramSketch ";" SketchStep

SketchStep   ::= PrimitiveOp SketchArgs              -- 单步原语调用
               | "copy" ";" FocusBlock                -- copy 遵守与 Program 相同的语法（§6.5）

SketchArgs   ::= (Literal | AttrRef | Computed | Hole)*
Hole         ::= "?" TypeHint?                        -- 未绑定参数孔位
TypeHint     ::= ":int" | ":color" | ":direction" | ":object_ref"
```

**与完整 Program 的关键区别**：

| 特性 | ProgramSketch（Layer 2 输出） | Program（Layer 4 求解） |
| --- | --- | --- |
| 控制流（ForEach / If） | **不含** — 草图只描述单步变换序列 | 允许 |
| Selector | **不含** — 对象选择由对齐方案隐式决定 | 显式 Selector |
| 参数孔位（Hole） | **允许** — `?` 表示尚未绑定的参数 | 不允许（全部绑定） |
| 序列复合 | 允许（多步原语 `;` 串联） | 允许 |

**语义说明**：

- **Hole**：Layer 2 根据 diff 可以确定某步需要 `translate`，但偏移量可能无法从单个 pair 中唯一确定。此时生成 `translate(?, ?:direction)` 留待 Layer 3 在跨 pair 一致性检查中绑定或淘汰。
- **无控制流**：Layer 2 生成的是针对单个对齐 pair 内的单步变换描述。控制流（对哪些对象做哪些条件分支）由 Layer 3 在组装假设时构造——当多个 SketchStep 需要按条件分派到不同对象子集时，Layer 3 将其提升为包含 ForEach/If 的完整 Program。
- **ProgramSketch → Program 的提升发生在 Layer 3→4 的边界**：Layer 3 筛选出的假设中，`program` 字段已是完整 Program（所有 Hole 已绑定，必要的控制流已插入）。Layer 4 直接执行，不再处理孔位。

## 5. Layer 3：任务相关结构筛选层

### 5.1 职责

这是整个系统最核心的模块。从 Layer 2 输出的大量候选中，选出当前任务真正需要的**最小充分结构**——最少的对象分割 + 最简的变换程序 + 最少的约束，使得所有训练示例都能被解释。

### 5.2 假设的定义与筛选目标

**假设（Hypothesis）的正式定义**：Layer 3 的基本搜索单元是一个五元组：

```text
Hypothesis = (plan_id, alignment_id, alignment_family_id, constraint_subset, program)
```

- `plan_id`：选定的分割方案；
- `alignment_id`：代表性原始对齐方案 ID，用于 provenance 与归因；
- `alignment_family_id`：该分割方案下真正用于绑定程序与约束的 family 级对齐键；
- `constraint_subset`：从该对齐家族诱导出的 `candidate_constraints` 中选取的约束子集（所有约束的 `alignment_family_id` 必须与假设的 `alignment_family_id` 一致）。约束子集内部区分两类：
  - **强约束**：在所有 train pair 上均成立的约束（`holds_in` 覆盖全部示例）。筛选阶段必须保留，test 阶段违反则直接淘汰。
  - **弱约束**：在多数但非全部 train pair 上成立的约束（`holds_in` 覆盖 $\geq 50\%$ 示例且 $< 100\%$）。筛选阶段不参与硬筛选，但保留在假设中供 test tie-break 使用。
  弱约束的存在不影响筛选期间的 MDL 评分（仅强约束计入 $L(h)$），但为 test 阶段提供额外的区分度；
- `program`：从该对齐家族诱导出的 `candidate_transforms` 中组合而成的 DSL 程序（所有变换的 `alignment_family_id` 必须与假设的 `alignment_family_id` 一致）。

Step 1 实现注记：当前 Step 1 代码已经把 `alignment_id` 与 `alignment_family_id` 作为独立字段显式保留，不再让 `alignment_id` 承载 family 级语义。

**绑定规则**：同一假设内部，`program` 和 `constraint_subset` 必须来自同一个 `alignment_family_id`。禁止"程序来自对齐家族 A、约束来自对齐家族 B"的混搭——这类混搭在结构上没有语义保证，是实现漂移的典型来源。

筛选层要同时回答三个问题：

1. **选哪个分割方案和对齐家族？** 从多个候选方案中选择一个最优 `(plan_id, alignment_family_id)` 对。（跨方案混合——从不同方案中各取部分对象——作为预留能力，Step 1-2 不实现，原因是需要统一对象命名空间和 plan-composition 层。）
2. **选哪些变换？** 从该对齐方案诱导的候选变换中选出能一致解释所有训练示例的最简变换程序。
3. **选哪些约束？** 从该对齐方案诱导的候选约束中构造约束包 `(strong_core, weak_context)`。其中 `strong_core` 是针对当前候选池的**贪心最小强约束核心**：在不改变 train 上完全匹配候选集合的前提下，尽可能移除冗余强约束；`weak_context` 为全部弱约束，仅用于 test tie-break，不纳入最小化目标。

### 5.3 评分框架

采用两阶段评分，取 MDL（最小描述长度）风味：

**第一阶段：精确筛选。** 只保留在**所有**训练示例上都达到 100% 像素匹配的假设。在这些完全有效假设中，按描述长度 $L(h)$ 升序排列，选最简者。

**第二阶段：近似回退。** 若没有假设达到全部 100% 匹配，则按以下公式排序：

$$\text{Score}(h) = \sum_{i} \text{mismatch}_i(h) + \lambda \cdot L(h) \qquad (\text{越小越好})$$

其中：

- $\text{mismatch}_i(h)$ 是假设 $h$ 在第 $i$ 个训练示例上的**不匹配像素数**；
- $L(h)$ 是假设 $h$ 的**描述长度**（简洁性）；
- $\lambda$ 是权衡系数（初始设为 1.0）。

选 Score 最低的假设，并标记为"不完全解释"。

使用不匹配像素数而非匹配率的原因：匹配率在 $-\log$ 下区分度很差（0.95 和 0.50 的差异仅为 0.05 vs 0.69），而像素数差异直接反映错误规模，更适合做排序依据。

#### 5.3.1 解释力的操作化定义

解释力 = 假设 $h$（分割方案 + 变换程序 + 约束集）应用于每个 train input 后，所产出网格与 train output 的**逐像素比较**。

- 完全匹配（不匹配数 = 0）：该假设完美解释该示例。
- 部分匹配：记录不匹配像素数和不匹配区域，供 Layer 5 归因使用。
- 第一阶段评分只考虑完全匹配的假设；第二阶段评分使用不匹配像素数之和。

#### 5.3.2 简洁性的操作化定义

描述长度 $L(h)$ 定义为假设 $h$ 在 DSL 中的编码长度：

- 使用的分割方案数量（每个分割方案 +1）
- 变换程序的 AST 节点数（每个节点 +1）
- **强约束**的数量（每个强约束 +1；弱约束不计入 $L(h)$）
- 变换中引用的具体常量数量（每个常量 +1；引用对象属性的不计入）

**弱约束不计费的理由**：弱约束是“有可能有用但不确定”的附加信息，用于 test tie-break 的区分度，而非假设的核心结构。若弱约束也计入 $L(h)$，会导致筛选偏好“约束少”的假设，而不是“结构简洁”的假设——这是两个不同的概念。

#### 5.3.3 跨示例一致性

一个假设必须在**所有**训练示例上使用**同一个程序结构**，只允许以下差异：

- 对象实例不同（因为每个示例的网格内容不同）；
- 参数值不同，但参数**来源**相同（例如"平移距离 = 对象宽度"，不同示例宽度不同但规则相同）。

若一个假设在不同示例上需要不同的程序结构，则该假设的一致性得分为零。

### 5.4 等价类归并

在 MDL 评分之前，先通过**测试诱导等价**对候选假设去重。

**定义**：两个假设 $h_1, h_2$ 在当前训练示例集 $\{(I_i, O_i)\}$ 上**测试等价**，当且仅当对所有训练示例 $i$， $h_1(I_i) = h_2(I_i)$（逐像素相等）。

**归并过程**：

1. 在所有 train pair 上执行全部通过一致性检查的候选假设，记录每个假设的输出向量 $(h(I_1), h(I_2), \ldots)$；
2. 按输出向量哈希分组，所有输出完全相同的假设归入同一等价类；
3. 每个等价类内，取描述长度 $L(h)$ 最短的假设作为**代表元**；
4. 后续 MDL 评分和 beam search 仅在代表元之间进行。

**重要：等价类归并是评估缓存，而非语义删除。** 整个等价类的全部成员被保留在内存中，只是非代表元不参与 beam search 评估。

**Test 阶段的等价类决策规则**：当代表元被选为最优假设 $h^*$ 后，test 阶段不是简单地执行代表元——因为 test 没有 ground truth，系统无法判定代表元是否"失败"。正确的做法是**并行执行 + 显式 tie-break**：

1. **并行执行**：将 $h^*$ 所属等价类的所有成员在 test input 上执行；
2. **约束集统一**：所有等价类成员统一使用**代表元 $h^*$ 的约束集** $C^*$ 进行评估，而非各自的约束集。原因：等价类按 train 输出哈希分组，成员的约束集可能不同（数量和强度各异），若用各自约束集评估，"约束多但弱"的候选会天然压过"约束少但强"的候选，比较基准不一致。
3. **执行筛选**：过滤掉执行崩溃或违反 $C^*$ 中任何**强约束**（在所有 train pair 上均成立的约束）的候选；
4. **Tie-break 排序**（对剩余候选按以下优先级降序排列）：
   - 约束加权满足率： $\frac{\sum_{c \in C^*} w_c \cdot \mathbf{1}[c \text{ satisfied}]}{\sum_{c \in C^*} w_c}$（其中强约束权重 $w_c = 2$，弱约束 $w_c = 1$；满足率越高越好）；
   - 属性引用占比（`AttrRef` 参数占总参数的比例越高，泛化能力越强）；
   - 描述长度 $L(h)$（越短越好）。
5. 选排名第一的候选作为最终 test output。若所有成员均执行失败，标记该任务为未解。

这样做的原因是：训练集上行为等价的两个假设可能在 test 上表现不同（例如一个依赖硬编码常量、一个依赖属性引用，当 test 的常量值与 train 不同时后者才正确）。仅用代表元做搜索缩减，用整个等价类 + tie-break 做 test 鲁棒性保障。

在跨任务模式中（参见 §11），等价类归并还用于知识库层面的搜索缩减：代表元作为假设库的缓存键，但等价类成员在回归测试和迁移测试中保留可回退能力。

### 5.5 筛选算法

```text
输入：来自 Layer 2 的 CandidateSet 列表（每个分割方案一个）
输出：最优假设 h* = (plan_id, alignment_id, alignment_family_id, constraint_subset, program)

对每个 CandidateSet（对应一个 plan_id）：
  对每个 alignment family（对应一个 alignment_family_id）：
    a. 取该 alignment 诱导的候选变换（绑定键匹配），按 diff 匹配度排序；
    b. 先组装该 family 下全部合法 hypothesis；在当前 Step 1 实现中，再按轻量 `pre_priority` 取 top-K hypothesis 进入执行评估；
    c. 取该 alignment 诱导的候选约束（绑定键匹配），分为两类：
       - 强约束：holds_in 覆盖所有 train pair；
       - 弱约束：holds_in 覆盖 ≥50% 且 <100% 的 train pair；
       在强约束上做贪心最小化：
         strong_core = all strong constraints
         按谓词复杂度从高到低遍历 c in strong_core:
             若移除 c 后，当前候选池中“在所有 train pair 上 100% 匹配且通过一致性检查”的 program 集合不变：
                 从 strong_core 中删除 c
       weak_context = all weak constraints
       constraint_subset = {strong: strong_core, weak: weak_context}
    d. 对每个 (program, constraint_subset)，检查跨示例一致性（§5.3.3）；
    e. 组装假设 h = (plan_id, alignment_id, alignment_family_id, constraint_subset, program)；其中 `alignment_id` 用于原始 provenance，`alignment_family_id` 用于 family 级绑定；

0. 等价类归并：对全部假设按 §5.4 分组，以代表元参与后续评估（完整等价类保留备用）；
1. 对代表元假设，在所有 train pair 上执行并统计不匹配像素数；
2. 第一阶段筛选：收集所有示例均 100% 匹配的假设，按 L(h) 升序排列；
3. 若第一阶段有结果，选 L(h) 最小的假设 h*；
4. 否则进入第二阶段：跨所有 (plan_id, alignment_family_id)，
   选 Score(h) 最低的假设 h*，并标记为"不完全解释"。
```

**关键保证**：上述流程中，候选变换和候选约束在步骤 a/c 的筛选阶段即按同一个 `alignment_family_id` 过滤，确保同一假设内的所有组件来自同一个对齐家族；而 `alignment_id` 只保留原始 provenance，供调试和归因使用。

### 5.6 学习到的优先级模型

上述筛选是纯搜索过程，效率取决于候选排序的质量。这里引入第一阶段唯一允许的学习组件：

**利用 ConceptARC 的概念组训练一个候选排序模型。** 但它不是 Step 1-2 或 Step 3A 的前置依赖，而是 Step 3B 起启用的可选搜索加速器。

- 训练数据：**按正式增长阶段递增**——模型在每个阶段仅使用当前已解锁概念组的训练任务（每组前 6 个）。正式增长的 Stage 1：Copy + Center 共 12 任务；Stage k：累计前 k 个阶段的概念组训练任务；最终 Stage 7 覆盖全部 16 组 × 6 = 96 任务。Step 1-2 不训练该模型。
- 诊断数据：ConceptARC 每个概念组的后 4 个任务（16 组 × 4 = 64 任务），仅用于前向评测，不参与训练。
- 模型输入：Layer 1 输出的对象/关系特征摘要。
- 模型输出：已解锁概念组上的置信度分布（未解锁概念组的输出头冻结为零）。
- 作用：根据概念组预测，为对应变换原语子集和关系类型子集赋予高优先级，缩小有效搜索空间（软权重，非硬门控，详见 §10）。概念组软权重**直接来自当前排序模型 $M_t$ 的输出分布**，不额外维护一个随任务成败在线更新的 `concept_weights` 表。若模型未启用，则 Layer 3 回退到纯手工启发式预排序。
- **训练时机**：每个正式增长阶段结束时（增长循环完成后），用累计训练任务重新训练排序模型。这使符号知识层（ $K_t$ ）和学习先验层（排序模型）同步渐进，实验中可分离两者的增益贡献。

**数据分割纪律**：训练集和诊断集的划分在实验开始前固定，不允许事后调整。任何使用诊断集数据训练模型的行为都应视为违规。

**为什么不预训练后冻结？** 若排序模型从一开始就见过全部 16 组训练任务，则"渐进扩展"仅作用于符号知识库，而学习到的结构先验已提前全量可用。这会导致：(1) 无法区分搜索加速来自知识状态增长还是先验模型；(2) Step 1 的最小闭环验证实际上已隐含了全量概念信息，违反最小实验原则。

**排序模型的作用时机（保守设定）**：Step 1-2 与 Step 3A 都不启用排序模型，先验证“无学习加速”版本能否跑通，并先把 Step 2 显式移交的能力缺口处理成稳定的新基线。进入 Step 3B 后，由于未解锁输出头冻结为零，且 Phase A 使用阶段开始前的 $M_t$（本阶段新概念组的头尚未解锁），排序模型**无法在当前阶段帮助首次发现新概念**。这是刻意的保守设计，而非缺陷。排序模型的主要作用是**跨阶段迁移加速**：当 Stage k 引入的新概念组与之前已解锁的概念组共享原语子集时（如 Stage 3 的 MoveToBoundary 复用 Stage 1 已学到的 translate 偏好），排序模型通过已解锁头的联合预测缩小搜索空间。阶段内首次发现新概念，完全依赖系统主干（DSL 搜索 + MDL 筛选）。

这样系统主干是 **"结构搜索器 + 学习到的优先级模型"**，而非端到端预测器。ARC-AGI-2 用来测"最后能不能解"，ConceptARC 用来学"该优先怀疑哪些结构"。

## 6. Layer 4：求解层

### 6.1 职责

把 Layer 3 筛选出的假设（分割方案 + 变换程序 + 约束集）应用于 test input，生成 test output 网格。

### 6.2 核心内部表示：任务图

求解层的工作对象不是像素矩阵，而是**任务图（Task Graph）**。

```text
TaskGraph = {
  nodes: [ObjectNode],                      -- 对象节点
  edges: [(ObjectNodeID, ObjectNodeID, Relation)],  -- 关系边
  global_constraints: [Constraint],         -- 全局约束
  transform_program: Program,               -- 变换程序
}
```

每个 train pair 先变成 (input_graph, output_graph)。Layer 3 学到的是 input_graph → output_graph 的变换程序，而非像素映射。对 test input，先用 Layer 1 构建其 input_graph，再执行变换程序得到 output_graph，最后渲染回像素网格。

这样做有三个好处：

1. **可解释**：失败可以定位到图变换的哪一步出错；
2. **可组合**：对象-态射-复合的结构天然兼容范畴论视角；
3. **可扩展**：未来换感知层时，中间结构层不必推倒重来。

### 6.3 两条求解路径

#### 路径 A：程序执行

当 Layer 3 已找到 100% 匹配的变换程序时，直接在 test input 的任务图上执行该程序。

执行步骤：

1. 用 Layer 3 选定的分割方案对 test input 做对象分割；
2. 构建 test input_graph；
3. 逐步执行变换程序中的每个原语；
4. 将结果 output_graph 渲染回网格。

#### 路径 B：约束求解（研究方向，主线步骤中暂不实现）

当 Layer 3 未找到完美程序，但找到了强约束集时，将问题转化为约束满足问题（CSP）：

1. 用已知约束生成 output 网格上的约束方程；
2. 用回溯搜索或 SAT 求解器寻找满足所有约束的 output；
3. 若存在多解，用简洁性偏好选最简方案。

在路径 B 工程化完成后，两条路径不互斥：优先尝试路径 A，若路径 A 结果不满足约束集，回退到路径 B。

**Scope 说明**：路径 B 不作为 Step 1-4 主线计划的依赖项。当前未定义变量域（按像素展开则搜索空间爆炸，按对象展开则需先确定输出对象集来源）、约束语言到 SAT/CSP 的编译边界、以及求解器选型。当前主线只使用路径 A；路径 B 仅在程序执行路径稳定后，作为单独研究分支工程化。

### 6.4 渲染规则

将 output_graph 渲染回像素网格时，必须遵循明确的规则，否则渲染错误会被误归因为 EXECUTION_FAIL。

| 规则 | 说明 |
| --- | --- |
| 背景色确定 | 使用 Layer 1 识别的候选背景色填充整个输出网格，再逐层叠加对象 |
| 叠加顺序 | 按对象面积从大到小叠加（大对象先放，小对象后放，小对象覆盖大对象） |
| 重叠冲突 | 两个同面积对象重叠时，后写入的对象覆盖先写入的（即变换程序中排在后面的对象优先） |
| 越界处理 | 对象像素超出输出网格边界时，截断超出部分，不报错 |
| 空区域 | 未被任何对象覆盖的像素保持背景色 |

### 6.5 DSL 定义

求解层操作的核心是一个精简 DSL。以下为初始版本的语法骨架：

```text
Program     ::= Statement | Program ";" Statement
Statement   ::= ForEach Selector ":" Transform
              | If Predicate ":" Transform "else" Transform
              | Transform

Transform   ::= PrimitiveOp Args
              | Transform ";" Transform       -- 序列复合
              | "copy" ";" FocusBlock          -- copy 后必须跟焦点块

FocusBlock  ::= FocusClause
              | FocusClause ";" FocusClause    -- 可分别操作原件和副本

FocusClause ::= "on_copy:" Transform
              | "on_original:" Transform

PrimitiveOp ::= "translate" | "rotate" | "flip" | "scale"
              | "delete" | "fill" | "extend_to_boundary"
              | "recolor" | "merge" | "sort" | "group_by"
              | "crop" | "tile" | "construct_grid"
              | "partition"                   -- 预留，Step 3+ 按需激活
              -- 注意：copy 不在此枚举中。copy 有专用产生式（Transform ::= "copy" ";" FocusBlock），
              -- 裸 copy 在文法层面即不可达，无需运行时检查。

Selector    ::= "all_objects"
              | "objects_where" Predicate

Predicate   ::= AttrCheck | RelationCheck | "and" Predicate Predicate
              | "or" Predicate Predicate | "not" Predicate

AttrCheck   ::= Attr Comparator Value
RelationCheck ::= "has_relation" RelationType ObjectRef

Args        ::= Literal | AttrRef | Computed
AttrRef     ::= "self." Attr | "other." Attr   -- 引用对象属性而非硬编码值
Computed    ::= AttrRef ArithOp AttrRef
```

关键设计决策：

- **`copy` 必须跟焦点选择器。** `copy` 产出 `Object × Object`（原件 + 副本），通过 `on_copy:` / `on_original:` 解构。裸 `copy` 后不跟焦点块是语法错误（参见 §4.2 copy 语义说明）。"复制到某位置"写作 `copy ; on_copy: translate(right, self.width)`。
- **优先使用属性引用（`AttrRef`）而非硬编码常量（`Literal`）**，这是保证跨示例一致性的关键。例如"平移距离 = self.width"而非"平移距离 = 3"。
- DSL 不是一次性设计完成的。初始版本只支持上述 16 个原语，后续根据 Layer 5  的"原语不足"归因扩展。扩展流程由 §11 的渐进增长循环管理。

## 7. Layer 5：验证与归因层

### 7.1 职责

对 Layer 4 的候选解进行验证，并在失败时输出结构化的归因标签，使迭代改进有据可循。

### 7.2 验证流程

Layer 5 根据是否有真实输出，分为两种运行模式：

#### 模式 A：有真值评估（train pair / ConceptARC / ARC-AGI-2 training）

对每个候选解执行以下检查：

1. **像素精确度**：候选 output 与真实 output 的逐像素比较，输出不匹配像素数。
2. **约束满足度**：候选 output 是否满足 Layer 3 选定的所有约束。
3. **结构一致性**：候选 output 的对象图结构是否与预期一致。

此模式下 `success`、`pixel_accuracy`、`failure_type` 等字段完整填写。当 `success = true` 时，`failure_type = "NONE"`（无失败）。

#### 模式 B：无真值推理（运行时未提供 test ground truth）

当系统**运行时不可访问** test output ground truth 时（如外部盲评提交、未公开标签的竞赛协议），Layer 5 无法执行像素精确度比较。此模式下：

1. **约束满足度**：候选 output 是否满足从 train pair 提取的强约束集（这些约束不依赖 test 真值，可独立检查）。
2. **结构一致性**：候选 output 的对象图结构是否符合 train pair 归纳出的规律（如对象数量、颜色集合、尺寸关系）。
3. **自洽性检查**：变换程序在所有 train pair 上是否仍然 100% 匹配（确认程序本身没有退化）。

此模式下归因输出的字段调整如下：

| 字段 | 模式 A（有真值） | 模式 B（无真值） |
| --- | --- | --- |
| `success` | 像素精确匹配判定 | `null`（无法判定） |
| `pixel_accuracy` | 精确计算 | `null` |
| `failure_type` | `NONE`（成功）或四类归因 | 约束违反或执行崩溃 → `EXECUTION_FAIL`；其余 → `UNVERIFIABLE` |
| `failure_confidence` | 正常分级 | `EXECUTION_FAIL` 正常分级；`UNVERIFIABLE` → `null`（不适用） |
| `constraints_satisfied/violated` | 正常填写 | 正常填写（不依赖真值） |
| `selected_*` 字段 | 正常填写 | 正常填写 |

**模式选择规则**：模式选择取决于**运行时是否向系统提供 test output ground truth**，而非数据集分区本身。同一数据集在不同运行配置下可使用不同模式。

| 运行场景 | 模式 | 说明 |
| --- | --- | --- |
| 本地开发/调试（任何数据集，有真值） | A | 所有 JSON 含 `test.output` 且系统可访问 |
| Step 1-2 全程 | A | ConceptARC 所有任务均有真值 |
| Step 4 本地评估（ARC-AGI-2，有真值） | A | 仓库中 training/evaluation JSON 均含 `test.output` |
| Step 4 外部盲评提交（真值不可访问） | B | 竞赛协议下系统无法读取 `test.output` |
| 消融实验（人为遮蔽真值） | B | 用于测试模式 B 自身的约束检查 / 自洽性检查质量 |

**关键区分**：当前仓库的 ARC-AGI-2 evaluation JSON 包含 `test.output`，因此本地运行时可使用模式 A 获得完整归因。"ARC-AGI-2 evaluation"不等同于"无真值"——仅当外部盲评协议禁止访问答案时，才切换到模式 B。

### 7.3 四类失败归因与子类型

与研究计划第 10 节一致，失败归因分为四个**主类型**：

| 归因类型 | 标签 | 判断依据 |
| --- | --- | --- |
| 没发现结构 | PERCEPTION_FAIL | Layer 1 的所有分割方案中都没有包含关键对象 |
| 结构选错了 | SELECTION_FAIL | 关键对象存在于某个方案中，但 Layer 3 没有选中该方案/变换 |
| 结构没保住 | ABSTRACTION_FAIL | Layer 3 选对了结构，但 DSL 表达力不足以编码正确变换 |
| 推理执行失败 | EXECUTION_FAIL | 程序正确但执行过程中出现边界情况或参数错误 |

#### 7.3.0 SELECTION_FAIL 的结构化子类型

由于假设已定义为五字段结构 `(plan_id, alignment_id, alignment_family_id, constraint_subset, program)`，SELECTION_FAIL 需进一步细分为结构化子类型，以精确定位失败环节。这是**硬接口约束**——归因输出的 `failure_detail` 字段在 `failure_type = SELECTION_FAIL` 时，必须填入以下枚举值之一，不允许自由文本：

| 子类型 | 含义 | 判断依据 |
| --- | --- | --- |
| `PLAN_ERROR` | 选错了分割方案 | 正确对象在其他 plan 中存在但当前 plan 中缺失 |
| `ALIGNMENT_ERROR` | 分割方案正确，但选错了对齐方案 | 正确对象对齐在其他 alignment 中存在但当前 alignment 的 matched_pairs 将不相关对象配对 |
| `CONSTRAINT_SUBSET_ERROR` | 对齐正确，但约束子集不足或过强导致正确程序被排除 | 正确程序在放宽/收紧约束后可通过 |
| `PROGRAM_ERROR` | 对齐和约束均正确，但选错了变换程序 | beam 中存在更优程序但被 pre_priority 排挤 |

**其他三个主类型**（PERCEPTION_FAIL、ABSTRACTION_FAIL、EXECUTION_FAIL）的 `failure_detail` 保持自由文本，因为它们的失败定位已经足够明确（分别对应 Layer 1、DSL 表达力、程序执行），无需进一步结构化。

#### 7.3.1 归因置信度分级

由于系统使用 beam search（top-K 截断）和概念组软权重等剪枝机制，归因标签**不总是可以严格判定**的。例如，ABSTRACTION_FAIL 可能实际是 SELECTION_FAIL（正确程序被 beam 截断排除），PERCEPTION_FAIL 可能实际是概念组权重导致正确原语被低优先级排挤。

因此归因标签分为两个置信度层级：

| 层级 | 标签后缀 | 含义 |
| --- | --- | --- |
| 可证实 | `_CONFIRMED` | 有确凿证据支持该归因，不因搜索截断而改变 |
| 推测 | `_SUSPECTED` | 在当前搜索配置下的最佳推断，但无法排除截断/剪枝导致的误归因 |

**可证实的判定条件**：

- **PERCEPTION_FAIL_CONFIRMED**：人工检查或穷举验证 Layer 1 所有方案均不包含关键对象（对象在原始网格中确实不可分割出来）。
- **EXECUTION_FAIL_CONFIRMED**：程序在 train pair 上 100% 匹配但在 test 上不匹配，且程序确实被执行了（非截断）。
- **ABSTRACTION_FAIL_CONFIRMED**：穷举当前 DSL 在有限组合深度内的所有合法程序，均无法匹配——仅当组合空间可穷举时才可判定。

**其余情况**均标记为 `_SUSPECTED`。实际中大多数归因将是 `_SUSPECTED`。

### 7.4 归因输出格式

```text
Attribution = {
  task_id: str,
  eval_mode: "A" | "B",                    -- 评估模式（A=有真值, B=无真值；参见 §7.2）
  success: bool | null,                     -- 模式 B 下为 null（无真值无法判定）
  pixel_accuracy: float | null,             -- 模式 B 下为 null
  constraints_satisfied: [str],
  constraints_violated: [str],
  failure_type: "NONE" | "PERCEPTION_FAIL" | "SELECTION_FAIL" | "ABSTRACTION_FAIL" | "EXECUTION_FAIL" | "UNVERIFIABLE",
                                            -- NONE=成功；模式 A 下 success=true → NONE，success=false → 四类失败之一；
                                            -- 模式 B 下：EXECUTION_FAIL 或 UNVERIFIABLE（不允许 null）
  failure_confidence: "CONFIRMED" | "SUSPECTED" | null,
  failure_detail: str | null,               -- SELECTION_FAIL 时为枚举值（§7.3.0）；其他类型为自由文本；模式 B 下可为 null
  selected_plan: str,                       -- 使用了哪个分割方案（= hypothesis.plan_id）
  selected_alignment: str,                  -- 使用了哪个代表性原始对齐 ID（= hypothesis.alignment_id）
  selected_alignment_family: str,           -- 使用了哪个 family 级对齐键（= hypothesis.alignment_family_id）
  selected_program: str,                    -- 使用了哪个变换程序（DSL 文本）
  selected_constraints: {                   -- 使用了哪些约束（= hypothesis.constraint_subset）
    strong: [str],                           -- 强约束 ID 列表
    weak: [str],                             -- 弱约束 ID 列表
  },
  system_state_version: {                   -- 求解时使用的系统状态版本（审计用）
    knowledge_state: int,                    -- K_t 的 t 值
    model_version: int,                      -- M_t 的 version 值
  },
  search_stats: {
    candidates_generated: int,
    candidates_evaluated: int,
    search_time_ms: int,
    beam_saturated: bool,                   -- beam 是否已满（若满，则存在截断风险）
    pruned_by_concept_weight: int,          -- 被概念组软权重降级排除的候选数
  },
  expansion_hint: {                         -- 归因驱动的扩展建议（§11 消费）
    missing_primitive: str | null,           -- 推断缺失的原语类型
    missing_relation: str | null,            -- 推断缺失的关系类型
    missing_segmentation: str | null,        -- 推断缺失的分割策略
    hint_confidence: "CONFIRMED" | "SUSPECTED",  -- 继承归因的置信度
  },
}
```

`selected_alignment`、`selected_alignment_family` 和 `selected_constraints` 使 Layer 5 的归因可以精确追溯到该五字段假设的各个绑定组件。例如，当归因为 SELECTION_FAIL 时，可以进一步区分"选错了对齐方案"、"选错了对齐家族"还是"对齐正确但约束不足"。`system_state_version` 记录求解时的系统快照版本，确保诊断报告和增长循环的可审计性。

`expansion_hint` 字段将 Layer 5 的归因与 §11 的知识状态扩展机制连接：当归因为 ABSTRACTION_FAIL 时，分析失败原因生成扩展建议，供渐进增长循环中的扩展步骤消费。

**归因置信度对扩展准入的影响**：§11.3 的扩展准入条件根据归因置信度差异化处理——`CONFIRMED` 归因仅需 2 个任务触发即可进入扩展准入评估，`SUSPECTED` 归因需 ≥ 4 个任务触发才进入评估，以降低因搜索截断引发的误扩展风险。

### 7.5 捷径检测

Layer 5 额外负责检测系统是否依赖捷径。具体检测方法：

| 捷径类型 | 检测方法 |
| --- | --- |
| 模板匹配 | 对 ConceptARC 某组任务做几何微扰（整体平移 N 格），检查成绩是否大幅下降 |
| 枚举投机 | 检查变换程序是否使用了硬编码常量而非属性引用 |
| 黑箱表示退化 | 检查中间表示是否可反序列化为可读的"对象 + 关系 + 变换" |
| 频率偏差依赖 | 对无关因素（背景色、输入尺寸）做扰动，检查成绩变化 |

## 8. 混合实现策略

整体架构是**启发式 + 搜索 + 有限学习**的混合，不是纯神经也不是纯枚举。

| 组件 | 实现方式 | 理由 |
| --- | --- | --- |
| 对象分割（Layer 1） | 纯启发式 | 离散网格世界中，显式算法接近完美且零训练成本 |
| 关系提取（Layer 1） | 纯启发式 | 同上 |
| 候选变换生成（Layer 2） | 枚举 + 类型剪枝 | 有限原语库 + 类型系统控制组合爆炸 |
| 候选排序（Layer 3） | 可选学习模型（Step 3B+） | ConceptARC 概念组作为训练信号；Step 1-2 与 Step 3A 回退到手工启发式预排序 |
| 结构筛选（Layer 3） | MDL 搜索 | 核心算法逻辑，不应是黑箱 |
| 程序执行（Layer 4） | 确定性解释器 | 变换程序必须精确执行 |
| 约束求解（Layer 4） | CSP / 回溯搜索 | 备用路径，处理程序未完全确定的情况 |
| 归因分析（Layer 5） | 规则 + 统计 | 归因逻辑必须可审计 |

唯一的学习组件是 Layer 3 的候选排序模型。这是刻意为之：第一阶段要证明的是**结构发现与筛选机制本身的价值**，不是证明某个神经网络架构的优越性。若在第一阶段引入过多学习组件，将无法区分"成功是因为结构方法"还是"因为神经网络的拟合能力"。

## 10. 搜索空间控制

搜索爆炸是第一阶段最大的工程风险。以下机制分层控制搜索空间：

| 层级 | 控制机制 |
| --- | --- |
| Layer 1 | 分割方案数量硬限：初始 7 种策略（参见 §3.3—§3.5：cc4, cc8, bbox, bg_fg, repeat, nested, whole_grid） |
| Layer 2 | 原语库硬限：初始 16 个原语；组合深度硬限：最大 3 层 |
| Layer 2 | 类型系统剪枝：类型不兼容的组合不生成 |
| Layer 3 | 概念组软权重：仅在 Step 3B+ 启用；top-3 预测概念组的原语获得正常优先级，其余原语降至低优先级但不排除（详见下方两级评估机制说明） |
| Layer 3 | 两级评估：按廉价 `pre_priority` 预排序取 top-K hypothesis 进入执行评估；当前 Step 1 实现使用 `STEP1_BEAM_SIZE = 64`，最终假设选择仍用纯 `Score(h)` |
| Layer 4 | 执行预算：Step 1-2 先记录执行时长，不把 1 秒写成硬中断；Step 3 起再根据 profiling 结果固化超时阈值 |
| 全局 | 任务预算：60 秒作为目标量级而非 Step 1-2 的硬性 SLA，待原型稳定后再收紧 |
| 知识库 | 等价类归并：跨任务行为一致的假设以代表元做评估缓存 |
| 知识库 | 三阶段重写：每轮扩展后触发假设折叠 + 原语去重 + 死知识冻结 |

**概念组软权重与两级评估机制说明**：概念组优先级模型的输出是 16 个概念组的置信度分布，用作候选变换的**预排序权重**，而非最终评分的一部分。系统采用两级评估以真正减少执行评估成本：

**第一级：廉价预排序（不执行候选）。** 对 Layer 2 产生的候选变换，在执行前先用结构特征计算预排序分数：

$$
\text{pre\_priority}(h) = w_1 \cdot \text{concept\_match}(h) + w_2 \cdot \text{attr\_ref\_ratio}(h) - w_3 \cdot \text{ast\_depth}(h)
$$

其中：

- `concept_match(h)`：候选使用的原语子集与 top-3 预测概念组的匹配度（0~1），由概念组模型直接给出；
- `attr_ref_ratio(h)`：程序草图中 `AttrRef` 参数占总参数的比例（越高越可能泛化）；
- `ast_depth(h)`：程序 AST 深度（越浅越简洁）；
- 初始权重 $w_1 = 3, w_2 = 2, w_3 = 1$。

若排序模型尚未启用（Step 1-2），则令 `concept_match(h) = 0`，只保留 `attr_ref_ratio` 与 `ast_depth` 两项。当前 Step 1 实现按轻量 `pre_priority` proxy 预排序，只取前 `STEP1_BEAM_SIZE = 64` 个 hypothesis 进入第二级评估。**这一步不需要在 train pair 上执行候选**，仅基于程序结构特征，计算成本为 $O(1)$ 每个候选。

对当前 Step 1 收尾实现，上述两个结构特征采用保守 proxy，而不是完整 AST 语义分析：`attr_ref_ratio` 由 program 文本中 `target=`、`color=` 及符号参数（`input_width`、`input_height`、`object_width`、`object_height`）的计数相对 token 总数近似，`ast_depth` 由分号数量 `;` 减去 copy block 结构性分隔符（`on_copy:`、`on_original:`）后加 1 近似。beam 的当前配置值是 `STEP1_BEAM_SIZE = 64`；也就是说，Step 1 真实行为是"先按这一轻量 proxy 预排序，再取前 64 个 hypothesis 进入执行评估"。后续若进入 Step 2/3，可再把这套 proxy 升级成真正的 AST / 参数类型分析。

**第二级：执行评估 + 最终选择。** 对进入 beam 的 K 个候选，在所有 train pair 上执行并计算纯 `Score(h)`：

$$\text{Score}(h) = \sum_{i} \text{mismatch}_i(h) + \lambda \cdot L(h)$$

概念组先验**不参与** `Score(h)` 的计算。最终选 $h^*$ 仅由 `Score(h)` 决定。

这样概念组模型在预排序阶段就起到了搜索缩减作用——大量不相关的候选在执行前即被过滤，而非"全部执行完再重排"。同时最终假设选择完全基于客观解释质量，不受概念先验影响。

## 11. 知识状态与渐进增长

v0.2 及之前描述的五层架构是"单任务流水线"——每个任务独立求解，不在任务之间积累知识。这在 Step 1-2（最小闭环验证 + 单任务加固）中足够，但面对 ConceptARC 16 个概念组和 ARC-AGI-2 上千任务时，会产生两个问题：

1. **重复搜索**：相似类型的任务反复从零开始枚举相同的候选变换。
2. **知识堆积**：随着原语库扩展和关系类型增加，搜索空间线性膨胀，缺乏压缩机制。

本节引入**知识状态**和**渐进增长循环**，使系统在任务序列中逐步积累和精炼结构知识。该机制**从 Step 3B 起启用**；Step 1-2 与 Step 3A 都不启用阶段内写回、扩展准入或知识重写。

### 11.1 知识状态定义

系统在时刻 $t$ 维护一个**系统状态** $S_t = (K_t, M_t)$，由两部分构成：

- **知识状态 $K_t$**：符号层面的结构知识（原语、关系、假设库等）。
- **排序模型状态 $M_t$**：学习先验层面的概念组排序模型（参数 + 已解锁输出头集合）。

两者分别代表系统的"结构记忆"和"经验直觉"，在渐进增长循环中同步更新。

```text
SystemState = (K_t, M_t)

KnowledgeState = {
  primitives: PrimitiveRegistry,            -- 当前原语集（初始 16 个）
  relations: RelationRegistry,              -- 当前关系类型集（初始 7 种）
  segmentations: SegmentationRegistry,      -- 当前分割策略集（初始 7 种，参见 §3.5 method 枚举）
  hypothesis_library: [StoredHypothesis],    -- 已解任务的成功假设库
  expansion_log: [ExpansionEvent],           -- 扩展历史
}

ModelState = {
  parameters: Tensor,                       -- 排序模型参数
  unlocked_heads: set[ConceptGroup],        -- 已解锁的概念组输出头
  training_tasks: [str],                    -- 训练该模型版本时使用的任务 ID 集合
  version: int,                             -- 模型版本号（每次重训 +1）
}

StoredHypothesis = {
  task_ids: [str],                           -- 该假设成功求解的任务集
  program: Program,                          -- DSL 程序
  concept_group: str | null,                 -- 所属概念组（若可判定）
  description_length: int,                   -- L(h)
  structural_signature: str,                 -- 程序结构的哈希（用于等价类归并）
}
```

初始状态 $K_0$ 由 §4.2 的原语集、§3.4 的关系集、§3.3 的分割策略构成，假设库为空。初始模型状态 $M_0$ 为禁用状态：参数可为空、已解锁输出头为空集。进入 Step 3B 的正式增长实验后，Stage 1 结束时训练得到 $M_1$，解锁 Copy + Center 两个头。

### 11.2 渐进增长主循环

系统以**阶段**为最外层单位组织任务序列，每个阶段包含一个或多个概念组。阶段内先完成全部概念组的训练任务，再统一执行扩展、重写、重训模型和诊断评测。这样 §11.7 的学习序列表与主循环的更新粒度保持一致。以下主循环仅在 Step 3B 的正式增长实验中启用。

```text
初始状态 S_0 = (K_0, M_0)
  K_0 = {最小原语集, 空假设库}
  M_0 = {随机参数, 空解锁头集, 空训练任务集, version=0}

for stage in learning_stages:
    H_stage = []        -- 本阶段新解出的假设缓冲区，不在 Phase A 内回写到 K_t
    T_stage = set()     -- 本阶段成功求解的任务 ID 集合

    -- Phase A: 求解（仅训练任务；使用当前 S_t）
    for concept_group in stage.concept_groups:
        for task in concept_group.train_tasks:
            1. 查询假设库：是否有已知假设可直接适用（结构匹配 + 参数重绑定）
            2. 若命中 → 尝试直接执行，成功则跳过搜索
            3. 若未命中 → 走完整五层流水线求解（排序模型使用 M_t）
            4. 成功的假设追加到 H_stage，并将 task_id 记录到 T_stage

    K_stage = merge_hypotheses(K_t, H_stage)
    solved_task_set_stage = solved_task_set(K_t) ∪ T_stage

    -- Phase B: 归因驱动的扩展
    收集本阶段所有失败归因中的 expansion_hint：
        - ABSTRACTION_FAIL → 候选新原语
        - PERCEPTION_FAIL → 候选新分割策略
        - SELECTION_FAIL + 特定模式 → 候选新关系类型
    对每个扩展候选执行准入评估（基于 K_stage 和 solved_task_set_stage，详见 §11.3）

    -- Phase C: 知识重写
    对阶段候选知识状态执行扩展与三阶段重写（详见 §11.4）
    K_{t+1} = rewrite(expand(K_stage, Δ_stage))

    -- Phase C': 重训排序模型
    用累计已解锁概念组的训练任务重新训练排序模型：
        M_{t+1}.unlocked_heads = M_t.unlocked_heads ∪ stage.concept_groups
        M_{t+1}.training_tasks = 所有已解锁阶段中的训练任务
        M_{t+1}.version = M_t.version + 1
    S_{t+1} = (K_{t+1}, M_{t+1})

    -- Phase D: 诊断评测（纯前向，不回写；使用完整 S_{t+1} = (K_{t+1}, M_{t+1})）
    for concept_group in stage.concept_groups:
        for task in concept_group.diagnostic_tasks:
            用 S_{t+1}（含重训后的排序模型 M_{t+1}）求解，记录结果
            不写入假设库、不触发扩展、不更新模型
```

关键约束：

- **Phase A 使用固定的 $S_t$。** 求解期间知识状态和排序模型均不变；成功假设只写入阶段缓冲区 `H_stage`，不在本阶段内回写到 `K_t`。这避免了求解过程中状态变动导致的不可复现性，也避免了阶段内任务顺序影响后续任务结果。
- **阶段候选状态在 Phase A 结束后才显式构造。** `K_stage = merge_hypotheses(K_t, H_stage)` 只用于本阶段后续的扩展准入、知识重写和回归验证；它不是 Phase A 内可见的在线状态。
- **Phase C' 在 Phase C 之后、Phase D 之前。** 这确保诊断评测反映的是完整的系统更新（知识 + 模型），而非混合状态。
- **Phase D 不影响 $S_t$。** 诊断任务的求解结果仅用于生成报告，不改变知识状态或模型状态的任何组件。
- **硬规则：诊断评测使用重训后模型。** Phase D 使用的排序模型版本 = $M_{t+1}$（本阶段重训后），而非 $M_t$（重训前）。诊断报告中需记录模型版本号以供审计。

### 11.3 扩展准入

不是所有归因建议都应被接受。扩展候选 $\Delta$ 必须通过以下准入条件：

这里的**已解任务集**定义为 `solved_task_set_stage = solved_task_set(K_t) ∪ T_stage`：既包含进入本阶段前已稳定写入 `K_t` 的历史已解任务，也包含本阶段刚解出并暂存于 `H_stage` 的任务。这样扩展准入不会只保护历史任务而破坏本阶段新获得的成功假设。

| 条件 | 判据 |
| --- | --- |
| 最小触发频次 | `CONFIRMED` 归因：≥ 2 个不同任务独立提出；`SUSPECTED` 归因：≥ 4 个不同任务独立提出（参见 §7.3.1） |
| 不破坏已解任务 | 扩展后，在 `solved_task_set_stage` 上重跑 `K_stage` 中的成功假设，结果不变 |
| 搜索空间增量可控 | 新原语引入的组合数增量 ≤ 当前总候选数的 20% |

准入检查由两步完成：

1. **廉价筛选**：统计触发频次（按置信度分级）+ 估算搜索空间增量（ $O(1)$ 计算），排除明显不值得的候选。
2. **回归验证**：对通过筛选的候选，在 `solved_task_set_stage` 上重跑 `K_stage` 中的成功假设，确认无退化。

通过准入的扩展进入本阶段的候选状态，并在 Phase C 完成后随 $K_{t+1}$ 一并正式落盘；同时记录到 `expansion_log`。

### 11.4 三阶段知识重写

每轮扩展后，对知识状态执行三阶段重写，将"规则堆积"转化为"结构升级"。

#### Pass A：假设等价类折叠

**目标**：合并假设库中行为等价但结构不同的假设。

**方法**：

1. 对假设库中的所有假设，在其各自成功求解的任务集上计算输出向量；
2. 两个假设归入同一等价类的条件：在双方成功任务集的**并集**上逐一执行，输出完全相同。若一方在另一方的某个任务上执行失败或输出不同，则它们**不等价**（最多标记为"部分相似"，不归并）；
3. 每个等价类内选 $L(h)$ 最短的假设作为代表元，用于假设库命中查询和搜索加速；
4. 与任务层等价类（§5.4）一致，非代表元不删除，仅降低搜索优先级。迁移时使用与 §5.4 相同的并行执行 + tie-break 规则。

**为什么用并集而非交集**：若仅要求在交集（重叠任务）上一致，两个在不同任务域上语义完全不同的假设可能因少量重叠任务上的偶然一致而被合并，进而污染假设库命中、概念权重更新和迁移优先级。并集验证确保折叠后的等价类在所有已知场景上行为一致，是更保守但更安全的标准。

**效果**：随着任务数量增加，大量特化假设（如"红色对象右移 2 格"、"蓝色对象右移 3 格"）被更通用的假设（"∀obj: translate(obj, right, self.width)"）替代。但由于并集验证，只有真正在所有已知任务上可互替换的假设才会被折叠，避免了过度合并。

#### Pass B：原语别名消除

**目标**：识别并消除原语库中的功能冗余。

**方法**：

1. 扫描假设库中所有使用过的原语组合；
2. 若原语组合 $p_1 ; p_2$ 在假设库中每次出现都可被单个原语 $p_3$ 替换且行为不变，则记录此别名关系；
3. 但不立即删除——只在别名出现 ≥ 3 次后才将 $p_1 ; p_2$ 统一替换为 $p_3$，并从假设库中更新受影响的假设。

**合法性验证**：替换后在受影响假设的任务集上重跑，确认输出不变。

#### Pass C：死知识冻结

**目标**：降低长期不用的知识的搜索优先级。

**方法**：

1. 统计每个原语、关系类型、分割策略在最近 $N$ 个任务中的引用次数；
2. 引用次数为 0 的组件标记为"冻结"——不删除，但在候选生成时降至最低优先级；
3. 若被冻结组件在后续任务中重新被归因建议激活，则解冻并恢复优先级。

**效果**：搜索空间不会因为扩展而单调膨胀。随着系统对任务分布的"适应"加深，实际活跃的搜索空间趋于收敛。

### 11.5 三阶段重写的验证纪律

重写必须遵循严格的验证纪律，否则"优化"可能引入退化：

1. **重写只在求解阶段之外执行。** 不允许边求解边重写。
2. **每次重写后执行回归测试。** 在 `solved_task_set_stage` 上重跑阶段候选状态中的成功假设，确认所有成功假设仍然成功。
3. **回归测试不通过则整轮回滚。** 恢复到重写前的 `K_stage = merge_hypotheses(K_t, H_stage)`，并记录失败原因。
4. **重写日志完整记录。** 每个 pass 的输入、输出、删除/合并/冻结的具体对象，供人工审计。

### 11.6 知识状态用于加速求解

积累的知识状态通过以下机制加速后续任务的求解：

| 加速机制 | 作用层 | 说明 |
| --- | --- | --- |
| 假设库命中 | Layer 3 | 新任务先查已有假设，命中则跳过搜索直接执行 |
| 排序模型先验 | Layer 3 | 使用当前 $M_t$ 输出的概念组置信度分布为候选提供软优先级，缩小 beam search 的有效搜索空间 |
| 活跃组件优先 | Layer 2 | 候选生成时优先使用活跃（非冻结）原语和关系类型 |
| 假设模板迁移 | Layer 2 → Layer 3 | 将已解任务的假设泛化为模板（将常量替换为属性引用），作为新任务的初始候选 |

### 11.7 ConceptARC 概念组作为渐进扩展序列

ConceptARC 的 16 个概念组天然构成一个渐进扩展序列，类似于 LCOS-R 中机器人从家居域扩展到零售域、工业域的过程。每加入一组概念，都可能引入新的原语需求或新的关系依赖。

建议的学习序列（由简到难，前序组的知识有助于后续组）：

| 阶段 | 概念组 | 核心新增知识 |
| --- | --- | --- |
| 1 | Copy, Center | copy, translate 基础操作 |
| 2 | ExtractObjects, CleanUp | delete, 背景/前景分离 |
| 3 | MoveToBoundary, ExtendToBoundary | translate/extend 的边界参数化 |
| 4 | CompleteShape, FilledNotFilled | fill, conditional_map |
| 5 | HorizontalVertical, SameDifferent | rotate, flip, 对称关系 |
| 6 | Order, Count, TopBottom2D/3D | sort, group_by, 计数约束 |
| 7 | AboveBelow, InsideOutside | 复合关系 + 条件变换 |

每个阶段结束后执行一轮 Phase B（归因驱动扩展）+ Phase C（三阶段重写），使系统在进入下一阶段前完成知识整合。

这意味着：第 7 阶段结束时的 $S_7 = (K_7, M_7)$ 不是 $S_0$ 简单地叠加了 7 轮扩展，而是经过 7 轮"扩展 + 重编译 + 重训模型"后的完整系统状态；其中 $K_7$ 是精炼知识状态， $M_7$ 是与之同步的排序模型状态。

### 11.8 与 LCOS-R 的关系

本节的渐进增长设计直接借鉴了 LCOS-R（参考语义驱动的机器人语义重写框架）的核心思路。LCOS-R 在机器人语义编译领域验证了以下判断：

> 开放环境中的持续扩展，不仅依赖新语义的准入，更依赖新语义进入后对内部表示进行语义保持重写——否则就是规则堆积，而不是结构升级。

两个项目的对应关系：

| LCOS-R 概念 | structuralist-agi 对应 |
| --- | --- |
| 语义签名 $\Sigma_t$ | 原语集 + 关系类型表 + 假设库 |
| 有限可执行测试 $Q_t$ | 训练示例的 input→output 像素精确匹配 |
| 测试诱导等价 $\equiv_t$ | §5.4 等价类归并中的假设等价 |
| 三阶段 rewrite pass | §11.4 假设折叠 + 原语去重 + 死知识冻结 |
| 扩展 = 加新语义 + 重编译 | §11.2 的归因驱动扩展 + 知识重写 |
| 系统代价 $\text{Cost}(c)$ | MDL 描述长度 $L(h)$ |

关键差异：LCOS-R 面对真正的开放世界（新物体随时出现），ARC 的任务集预先给定。这使得我们可以精心设计学习序列（§11.7），在扩展节奏上拥有更多控制权。

### 11.9 与任务层 MDL 的同构关系

知识层的三阶段改写和任务层的 MDL 假设筛选，在结构上是同一种操作的两个实例：**给定不变量约束，搜索最小表示**。

| 维度 | 任务层（§5–§6） | 知识层（§11.4） |
| --- | --- | --- |
| 被压缩的对象 | 候选假设集 | 知识状态 $K_t$ |
| 保持不变的结构 | 单个任务的 input→output 映射 | 所有已解任务的可解性 |
| 压缩准则 | MDL 描述长度最短 | 回归测试全通过 + 表示最紧凑 |
| 去重手段 | 等价类归并（§5.4） | 假设折叠（Pass A） |
| 消除冗余 | 丢弃不匹配假设 | 原语别名消除（Pass B）+ 死知识冻结（Pass C） |

用范畴论语言：改写是知识范畴上的自函子 $R: \mathcal{K} \to \mathcal{K}$，将 $K_t$ 映射到更小的 $K_t'$，同时保持从 $K_t$ 到"已解任务集"的函子 $F$ 不变，即 $F \circ R = F$。这与任务层中 MDL 在保持 input→output 不变的前提下压缩假设描述长度，是同一个保结构映射原则——**用最少的结构解释最多的现象**。

这一同构目前仅作为设计观察记录，不改变现有架构。两层的触发频率、正确性判据、搜索对象差异较大（任务层每题触发、像素精确匹配、搜索 AST 程序空间；知识层每阶段触发、批量回归测试、搜索注册表条目），过早统一会导致不必要的抽象开销。

**可统一的条件**：当 Step 3 完成后，如果两层代码中出现明确可复用的模式——例如一个通用的 `compress(object, invariant, metric) -> minimal_object` 框架能同时服务于假设筛选和知识改写且无需 ad-hoc 分支——则值得提取公共接口。届时这一统一本身可以作为论文贡献点：结构保持压缩作为统一原则在任务求解与知识管理两个层次上的实例化。在此之前，保持分离实现、共享设计直觉即可。

## 12. 实施路线

不追求一开始覆盖全部任务，分五步逐层扩展。Step 1-2 先把单任务求解器做稳，Step 3A 先处理 Step 2 显式移交的残差能力缺口，再从 Step 3B 起启用知识增长与学习加速。

### 12.1 Step 1：最小闭环验证（单任务流水线）

**目标**：在 ConceptARC 的 Copy 和 Center 两个概念组的**训练任务**（每组前 6 个，共 12 个任务）上跑通端到端闭环。

**数据纪律**：Step 1 期间，Copy 和 Center 的诊断任务（每组后 4 个，共 8 个）**不可见**——不用于开发、调试或任何参数调整。这 8 个任务保留在诊断集中，供 Step 3B 的正式机制诊断报告使用。这确保了诊断集全程未在开发中暴露。

**作用域收缩**：Step 1 不追求覆盖文档中的全部能力，而只验证最小闭环是否成立。默认只启用 cc4 / cc8 / whole_grid 三种分割方案；只支持一对一对象对齐；原语收缩到 copy、translate、rotate、flip、delete、recolor、fill、crop 及有限的输出尺寸推断/渲染逻辑；不启用 merge / split / partition / group_by / tile / construct_grid；不启用约束求解路径 B。Step 1 允许的输出尺寸规则也只限于四类：保持输入尺寸、按变换后对象外接矩形扩张画布、裁剪到选中中心对象的 bbox、裁剪到 1x1 中心单元。

**知识状态**：Step 1 内部所有训练任务都使用固定的 $S_0 = (K_0, M_0)$ 独立求解，不启用阶段内写回，不启用知识重写，也不训练排序模型。本步的目的是验证五层流水线本身可以工作，而不是验证持续学习机制。

**验证内容**：

- Layer 1 能否正确分割出关键对象；
- Layer 2 的 copy/translate/fill/crop 等基础原语与有限输出尺寸推断是否足够；
- Layer 3 在无学习加速条件下的筛选是否有效（含 §5.4 等价类归并）；
- Layer 4 能否生成正确 output；
- Layer 5 能否给出有意义的基础归因标签。

**成功标准**：

- 主要标准：端到端闭环跑通——每一层的输入输出均产生合法结果，Layer 5 对每个任务输出结构化归因。
- ~~量化参考：12 个训练任务中精确求解数 ≥ 9；未解任务有明确归因标签且归因合理。~~ （早期草案估计，已由下方阶段性结论取代：实际结果 6/12，但闭环与归因链路已稳定。）
- 注意：具体数字在首次接触真实任务后可校准，但闭环跑通是硬性要求。若 Step 1 仍然依赖大量 ad-hoc 特判，则禁止进入 Step 2。

**阶段性结论（2026-04-04）**：当前 Step 1 的实际结果是：12 个训练任务上精确求解 6 个，其余 6 个主要归因为 `ABSTRACTION_FAIL`；同时，五层流水线、最小接口、pair/family 双层 ID 语义、最小 beam 与基础归因链路都已经稳定落地并通过测试。也就是说，Step 1 没有达到早期草案中的激进数值参考，但它已经充分证明“最小闭环可以工作”，且剩余问题主要集中在 Step 2 应处理的能力缺口，而不是架构性停摆。因此这里的工程决策应是：把 Step 1 作为已实施冻结的基线保存，并进入 Step 2 的单任务求解器加固，而不是因为当前结果未满额而判定路线失败或退出第一阶段。

### 12.2 Step 2：单任务求解器加固（仍不启用增长）

**目标**：在不引入知识增长和排序模型的前提下，把单任务求解器从 Copy / Center 扩展到一个**小而受控**的训练子集，优先覆盖与现有原语邻近的概念组，如 ExtractObjects、CleanUp、MoveToBoundary、ExtendToBoundary。

**数据纪律**：Step 2 仍只使用训练任务，不消费任何诊断集任务；新增概念组以小批量加入，目的不是追求覆盖率，而是稳定对齐、原语语义和执行链路。

**系统状态**：Step 2 仍使用固定的单任务流水线，不启用假设库写回，不启用扩展准入，不启用知识重写，不训练排序模型。Step 2 的产出是“稳定代码与明确边界”，而不是增长曲线。

**验证内容**：

- 对齐模块在平移、删除、简单颜色变化下是否稳定；
- 新增原语的语义是否足够明确，不再依赖样例级特判；
- 归因输出能否把“对齐错了”和“原语不够”区分开；
- 搜索时间、候选数、执行时长是否收敛到可用于 Step 3 设定真实 beam / timeout 的量级。

**成功标准**：

- 至少再增加若干相邻概念组后，系统仍保持可重复求解，而不是每加一组就重写架构；
- 新增原语和分割/对齐规则有明确的任务驱动依据；
- 已能据实设定 Step 3B 的 beam、候选上限和执行预算，而不是凭直觉估计；
- 若 Step 2 结束后仍无法稳定区分“对齐失败”和“程序表达力不足”，则不得进入 Step 3B 的正式增长实验。

**阶段性结论（2026-04-06）**：Step 2 已按“冻结边界 + 明确移交”的口径验收。当前稳定基线为：Step 2a 在 Copy + Center 上达到 **8/12**，按“有条件通过”接受；Step 2b 在新增四个概念组上达到 **MoveToBoundary 3/6、ExtendToBoundary 2/6（按冻结 7B 边界收口）、ExtractObjects 2/6、CleanUp 冻结为未收口能力缺口**；Phase 8 的 runner、概念组聚合、回归标记与诊断产物均已落地；Phase 9 的 36 任务全量回归结果为 **15/36 exact**，且已知成功样例无回退。

这意味着 Step 2 的工程目标“稳定代码与明确边界”已经满足，可以作为 Step 3A 的输入基线冻结保存。与此同时，Step 2 的遗留问题也已经完成归因并明确转交：Copy3/4 需要参数化 anchor-relative 位移语义，Copy5/6 需要 repeat / tile / construct_grid 一类构造语义，ETB1/3/5/6 需要几何条件化或非轴向扩张语义，CleanUp 需要超出当前 Step 2 红线的更强局部修复能力。它们都不再回流到 Step 2 内处理，而作为 Step 3A 的显式输入。

### 12.3 Step 3A：残差能力引入（仍不启用增长）

**目标**：在 Step 2 已验收冻结的单任务求解器基线之上，优先处理 Step 2 已明确移交的未通过任务，把“已知表达力缺口”转化为 Step 3 的第一批稳定能力，而不是一上来就把知识状态与排序模型接入主干。

**数据纪律**：Step 3A 仍以训练任务为主，不启用 ConceptARC 全量正式增长，也不消费诊断集做阶段性学习。优先使用 Step 2 已明确移交的残差任务族和最小必要的合成单测来验证新能力；诊断集继续保留给 Step 3B 的正式机制评测。

**系统状态**：Step 3A 仍使用固定的单任务流水线，不启用假设库写回，不启用扩展准入，不启用知识重写，也不训练排序模型。允许引入新的原语、参数化语义或更强选择/修复机制，但每一项都必须先在单任务层面完成“无回归 + 非 ad hoc + 边界清晰”的验证，再进入下一项。

**验证内容**：

- 参数化 anchor-relative 位移是否能稳定覆盖 Copy3/4 一类残差；
- repeat / tile / construct_grid 一类构造语义是否能覆盖 Copy5/6 一类输出尺寸变化任务；
- 几何条件化或非轴向扩张语义是否能覆盖 ETB1/3/5/6 这类冻结 7B 边界外残差；
- 若继续尝试 CleanUp，局部修复机制是否仍满足“不依赖任务名、颜色特判、图样特判”的红线；
- 每引入一类新能力后，Step 2 已验收通过的任务是否保持无回归。

**成功标准**：

- 至少引入 1-2 类能够真实消化 Step 2 显式残差的稳定新能力，而不是只增加更多搜索启发式；
- 新能力在对应残差任务族上产生可重复增益，且不破坏 Step 2 已通过任务；
- 每一项新能力都能被明确写成“可做什么、不能做什么”的冻结边界；
- Step 3A 结束时，形成一版新的稳定单任务基线，作为 Step 3B 的正式起点。

### 12.4 Step 3B：ConceptARC 正式增长实验（启用知识状态与排序模型）

**目标**：在 Step 3A 已经强化过的单任务求解器基线之上，正式启用 §11 的知识状态与渐进增长机制，并按 §11.7 的阶段序列扩展到 ConceptARC 全部 16 个概念组。

**数据纪律**：渐进增长循环（Phase A → B → C → C'）仅在 **96 个训练任务**（16 组 × 前 6 个）上执行。64 个诊断任务（16 组 × 后 4 个）做**纯前向评测**——使用当前 $S_{t+1} = (K_{t+1}, M_{t+1})$ 求解，但成功假设**不回写**假设库、不触发扩展准入、不更新模型。诊断任务的唯一用途是生成按概念组的机制诊断报告。

**系统状态**：Step 3B 使用一套**重新清零的正式实验状态**启动：代码、原语语义、分割规则沿用 Step 3A 结束后已经冻结的稳定版本，但假设库从空集开始，排序模型从禁用态 $M_0$ 开始。Stage 1 先在 Copy + Center 上运行无模型增长；Stage 1 结束后再训练得到 $M_1$，从 Stage 2 起把排序模型作为搜索加速器接入。

**验证内容**：

- 渐进增长机制是否产生了可度量的搜索加速（对比：同一任务集上使用 $S_0$ vs 使用 $S_t$ 的求解时间和成功率；若需拆分来源，再补做固定 $M$ 仅替换 $K$ 与固定 $K$ 仅替换 $M$ 的消融）；
- 三阶段重写是否有效压缩了假设库（跟踪假设库大小随阶段的变化曲线）；
- 原语库是否需要扩展，扩展了多少，扩展是否由归因驱动（而非手工指定）；
- 概念组排序模型是否提供了有意义的搜索加速（增量训练 vs 全量预训练冻结的对比实验）；
- 跨概念组的归因分布是否揭示系统性缺陷；
- **诊断集 vs 训练集的求解率差异**：若差异过大（如训练 > 80% 而诊断 < 50%），说明系统过拟合训练集结构。

**成功标准**：

- 训练集（96 任务）：≥ 65 个精确求解。
- 诊断集（64 任务）：≥ 35 个精确求解（纯前向，未参与增长）。
- 能明确指出哪些概念组最难以及为什么。
- 知识重写后的 $K_7$ 的假设库规模 < 7 轮扩展的假设库简单累加总量（即重写确实产生了压缩效果）。

### 12.5 Step 4：ARC-AGI-2（知识迁移验证）

**目标**：在 ARC-AGI-2 训练集的 1000 个任务上评估，并在 120 个评测任务上报告最终结果。

**主协议（冻结迁移评测）**：从 Step 3 结束时的**冻结**系统状态 $S_7 = (K_7, M_7)$ 启动。Step 4 的主结果一律采用纯前向协议：在 ARC-AGI-2 training 和 evaluation 上都**不回写假设库、不触发扩展、不执行知识重写、不更新排序模型**。这样报告的结果才能被解释为“ConceptARC → ARC-AGI-2 的知识迁移效果”，而不是“在 ARC 上继续在线学习后的效果”。

**补充协议（可选，单独汇报）**：若后续研究需要评估跨 benchmark 的持续学习能力，可在 ARC-AGI-2 training 上额外运行“继续增长”协议；但这必须作为独立实验汇报，且不得与主协议结果混合。

**验证内容**：

- 从 ConceptARC 积累的冻结系统状态 $S_7$ 在 ARC-AGI-2 上是否迁移有效；
- 假设库命中率和假设模板迁移成功率；
- 搜索空间控制是否仍然可行（知识库规模 vs 搜索效率的关系）；
- 冻结迁移协议与可选继续增长协议之间的性能差异（若运行补充实验）；
- 与三类基线的对比结果。

**成功标准**：由研究计划第 12 节定义的多指标联合判断。主指标采用冻结迁移协议下的结果。额外指标： $S_7$ 启动 vs $S_0$ 启动的求解率差异，量化累积系统状态的价值；若需要隔离符号层与排序模型的贡献，再补做 $K_7 + M_0$ 与 $K_0 + M_7$ 的消融。若运行继续增长协议，其结果只作为补充分析，不计入主结果。

## 13. 与范畴论视角的接口

第一阶段不要求完整的范畴论形式化，但架构设计应为后续接入留好接口：

| 架构概念 | 范畴论对应物 | 第一阶段要求 |
| --- | --- | --- |
| Object | 范畴中的对象 | 对象必须有明确的同一性和属性 |
| Relation | 态射 | 关系必须可复合（邻接 + 包含 → 路径） |
| Transform | 函子 | 变换必须保持指定关系（保结构映射） |
| Constraint | 交换图/泛性质 | 约束必须可验证 |
| 变换的顺序复合 | 态射复合 | 复合必须满足结合律 |

只要任务图的数据结构满足上述要求，后续范畴论形式化就只是加一层数学包装，而非架构重写。

## 14. 设计底线

以下六条是整个架构的不可妥协底线。违反任何一条意味着架构退化为黑箱。

1. **Layer 1 的输出必须是显式对象和关系，而非 embedding。**
2. **Layer 3 的筛选逻辑必须可审计，而非神经网络打分器。**
3. **Layer 4 的变换程序必须可打印、可阅读、可手动执行。**
4. **Layer 5 的归因字段 `failure_type` 必须为六个显式枚举值之一（`NONE` + 四类失败 + `UNVERIFIABLE`），不允许 `null` 或自由文本。** 模式 A 下：成功 → `NONE`，失败 → 四类之一；模式 B 下无法判定时使用 `UNVERIFIABLE`，而非留空。
5. **唯一允许的学习组件是 Layer 3 的候选排序模型，且该模型的输出是概念组置信度分布，而非直接给出答案。**
6. **知识重写只做保行为的压缩，不做改变已解任务结果的"优化"。** 每轮重写后的回归测试是不可跳过的硬性要求。
